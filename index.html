Design System Specimen

Fresh Greens

Tokens, type ramp, and design rules for the thesis navigation/safety app. The application is iPhone-first React Native; this specimen presents the tokens at parity with their in-app rendering.

Generated from theme/colors.ts and theme/typography.ts. Figma source maintained internally; private identifiers are intentionally omitted from this published specimen.

Colors

The palette splits into brand greens (used freely), reserved colors (UI safety signals only, with documented exceptions), neutrals, and a set of iOS-derived semantic tokens for labels, backgrounds, fills, and separators. Translucent values are shown over a checker pattern so the alpha channel is visible.

Typography

Naming follows Apple's iOS Human Interface Guidelines type ramp (Apple Inc., n.d.) — largeTitle / title1 / body / subheadline / footnote / caption1 / caption2 — with Regular and Emphasized weight variants. Same convention Figma uses, so the mapping between design and code is one-to-one. Weights follow React Native's standard scale (400 / 600 / 700); Apple's "Semibold" PostScript weight (590) maps to 600.

Design rules

The system isn't just tokens — it's the rules that govern when and where each token is allowed. These rules live alongside the code in .cursorrules at the repository root; below is a summary of the principles that shape every screen.

  • Reserved-color rule

    Orange, red, yellow, and pink are reserved as UI safety signals. They carry semantic weight — hazard, alert, caution, TBD — and are not used for brand decoration. Documented exceptions: Welcome's orange-sky brand splash; the navy uniform on the Officer/Trooper illustration; the daylight-gradient route polyline; safety-zone fills; and the report-button alert icon. Anywhere else, reserved colors are off-limits.

  • Typography emotional-restraint exception

    Title1/Regular (28pt at weight 400) is permitted on emotionally-charged screens — the in-modal user-prompt register. The default for headers is the Emphasized weight; Regular is reserved for moments where a held question reads better than a directive. Used on /pulled-over's Trusted Contact phase ("You're not alone."), the armed-or-not prompt ("Are you armed?"), and the Trip Summary screen.

  • Modal padding rules

    Modal screens use paddingHorizontal: 16 for tab-grid and card-based modals (Safety, Pulled-Over, Armed-or-Not, Report). Static-content screens — the ones that are mostly long-form text and a single CTA — use paddingHorizontal: 32 (Onboarding, Permissions, Trusted Contact Setup, Settings). The wider gutter on static-content screens gives the copy room to breathe; the tighter gutter on grids fits more tappable density.

  • Tap targets — iOS HIG 44×44 minimum

    Every interactive element has at least a 44pt-by-44pt effective tap area, per Apple's iOS Human Interface Guidelines (Apple Inc., n.d.). Either the visual itself meets that bar, or hitSlop extends the tap area beyond the visual. The default hitSlop across modal dismiss patterns is 12pt, which makes a 24pt visual icon a 48pt tap target. The exception clause is genuinely-constrained dense rows where a 44pt visual would dominate the layout — the Welcome terms checkbox, for example, is 24pt visual with hitSlop: 20 for a 64pt effective area.

  • Responsive widths — never hardcode 374

    Widths derived from the 390pt iPhone-baseline (often 374, 326, or 351) fail on the wider Pro Max line (430pt+) by floating in the middle with too much edge gutter. The responsive pattern is alignSelf: 'stretch' plus a parent paddingHorizontal equal to the intended gutter — the element grows with the device while the gutter stays constant.

  • One identity glyph across surfaces

    The user is represented as a car — same iconography wherever they appear in the app: the avatar button on /home, the profile glyph in /menu's hero. Differentiated from the trusted-friend pin (also a car) by color: fadedgreen for the user's own car, freshgreen for the trusted contact. The shared metaphor is the thesis claim rendered visually — the app sees the user as a car-in-the- system, not a generic abstract user.

  • Legal-guidance copy from authoritative sources

    The "What to Do," "What to Have," "What to Say," and "What to Know" bullets in the Pulled Over flow are sourced from the American Civil Liberties Union's Know Your Rights: Stopped by Police guidance (American Civil Liberties Union, n.d.). Tightening for at-a-glance UI use was permitted; substantive rephrasing was not. The right-to-record and right-to-leave bullets added during fidelity audit 4 trace directly to the same ACLU source. Sourcing legal copy from an authoritative third party is a system rule, not a per-screen choice — substituting paraphrased legal advice would weaken the thesis claim that the app respects the lived legal situation of Black drivers.

  • Navigation UX anchored in user-familiar metaphors

    Where the app introduces novel safety surfaces (the Pulled Over consolidated modal, the Trusted Contact flow), the surrounding navigation chrome — search, route preview, settings menu, profile greeting — borrows from established navigation apps so users don't pay a learning tax on top of the safety-flow novelty. The Settings menu's profile-row greeting, single-row sections, and swipe-with-peek tile carousel reference Waze's pattern (Waze Mobile Ltd., n.d.); the search-bar register and route bottom-sheet reference Google Maps (Google LLC, n.d.). The thesis contribution is the safety-aware routing and post-stop guidance, not a reinvention of map navigation — anchoring the familiar parts is what makes the unfamiliar parts legible.
